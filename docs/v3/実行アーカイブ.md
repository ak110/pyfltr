# パート B: 実行アーカイブ

v3.0.0で追加した「全実行の詳細を自動保存する」仕組みの設計判断と実装結果。

## 目的

エージェント連携時にJSONL出力のsmart truncationで削られた情報やツール生出力を事後参照できるようにする。
後続の`show-run` / `list-runs`サブコマンド（パートE）およびMCP詳細参照ツール（パートF）が読み取り側となる。

## 保存先

XDG Base Directory準拠のユーザーキャッシュ配下に集約する。
プロジェクト内にキャッシュディレクトリを作らないことで、`.gitignore` 運用の負担を増やさず、複数プロジェクト横断での参照を可能にする。

`platformdirs.user_cache_dir("pyfltr")` で解決する。

- Linux: `~/.cache/pyfltr/`
- macOS: `~/Library/Caches/pyfltr/`
- Windows: `%LOCALAPPDATA%\pyfltr\Cache`

環境変数 `PYFLTR_CACHE_DIR` を優先する（テストや運用上の強制上書き用）。

## ディレクトリ構造

```text
<cache_root>/runs/<run_id>/meta.json
<cache_root>/runs/<run_id>/tools/<tool>/output.log
<cache_root>/runs/<run_id>/tools/<tool>/diagnostics.jsonl
<cache_root>/runs/<run_id>/tools/<tool>/tool.json
```

`meta.json` の主なフィールド:

- `run_id`: ULID
- `version`: pyfltrのバージョン
- `python` / `executable` / `platform` / `cwd`: 実行環境
- `argv`: 起動時の引数列
- `commands`: 実行対象コマンド列
- `files`: 対象ファイル数
- `started_at` / `finished_at`: ISO 8601 (UTC)
- `exit_code`: 終了コード（finalize後）

ツール単位の `tool.json` の主なフィールド:

- `tool` / `type` / `status` / `returncode`
- `files` / `elapsed` / `diagnostics` / `has_error`
- `commandline`: 実際に実行されたコマンドライン

`output.log` はツールの標準出力・標準エラーを結合した生文字列。
`diagnostics.jsonl` は `ErrorLocation` をJSONL形式で1件1行にシリアライズしたもの。

## run_id

ULID（26文字、Crockford Base32、タイムスタンプ由来で辞書順ソート可能）を採用する。
`python-ulid` パッケージで生成し、JSONLの `header` レコードに `run_id` フィールドとして含める。

```json
{"kind":"header","version":"3.0.0","run_id":"01HXYZ...","commands":[...],"files":12}
```

UUID v4ではなくULIDを採用した理由:

- タイムスタンプ由来で辞書順ソート = 時系列順ソートとなる。`list_runs` の実装・デバッグが簡潔になる
- 人が見たときに新旧の判別がしやすい
- 十分な衝突耐性 (48bit timestamp + 80bit random)

## 自動クリーンアップ

保存世代数・合計サイズ・保存期間の3軸で制御する。
いずれかの閾値を超過した時点で古い順（run_id昇順）に削除する。

既定値:

- `archive-max-runs`: 100
- `archive-max-size-mb`: 1024
- `archive-max-age-days`: 30

`pyproject.toml` で上書き可能。0以下を指定するとその軸の自動削除は無効化される。

クリーンアップは各実行冒頭で `ArchiveStore.cleanup()` を呼び出して同期実行する。
将来的に非同期化する選択肢は残すが、v3初版では単純な同期実行で開始する。

## 書き込み経路

アーカイブ書き込みはTUI経路・非TUI経路・JSONL stdout有無のいずれでも発生させる。
漏れを防ぐため、`execute_command` 戻り値受信直後の独立フックとして提供する。

実装:

- `run_pipeline` が `ArchiveStore` を生成、`start_run` でrun_idを採番
- `run_pipeline` が `_archive_hook` を組み立て、`cli.run_commands_with_cli` と `ui.run_commands_with_ui` の `archive_hook` 引数へ注入
- 各実行器はツール完了時（fixステージも含む）に `archive_hook(result)` を呼び出す
- `run_pipeline` 終端で `finalize_run` を呼び、`exit_code` / `finished_at` を書き込む

JSONL stdoutストリーミングの `on_result` とは独立した経路にすることで、どちらか一方に切り替えても他方が失われない。

## オプトアウト

既定で有効。ストレージ消費が問題になるケースに備えて以下の手段で無効化できる。

- `--no-archive` CLIオプション
- `archive = false` 設定

無効化時は `ArchiveStore` を生成せず、`archive_hook` も `None` のまま。
JSONLの `header` レコードから `run_id` フィールドも省略される。

## 却下した代替案

- オプトイン扱いにする: エージェント連携時のUXを損なう。既定有効 + 自動削除で肥大化を抑える
- プロジェクトローカル (`.pyfltr-cache/`) に併設する: `.gitignore` 運用負担が増え、複数プロジェクト横断の再参照が困難。XDG準拠に集約する方が一貫性がある
- `/tmp` に保存する: 再起動で消失、情報漏洩リスクが高い

## ErrorLocation との差分

アーカイブ用のdiagnosticシリアライズ（`_error_to_dict`）はLLM向け出力（`llm_output.py`）とは独立した最小構造とする。
`ErrorLocation` の全フィールドを保存することで、将来 `rule_url` などのフィールドが追加されても追従しやすい。
