## 作業再開手順

1. 作業ブランチを最新化する（masterで作業中）
2. ドキュメント更新のみ必要な場合:
    - `README.md` / `docs/guide/*` / `mkdocs.yml` / `CLAUDE.md` / `CHANGELOG.md`を順に更新する
    - ドキュメント更新後に`uv run mkdocs build --strict`で整合性を確認する
3. すべての変更が完了したらバージョンを`3.0.0`としてリリースする

## 確定済みの設計判断（再確認不要）

以下はユーザー合意済みまたはエージェント判断で確定しており、v3.0.0実装全体で踏襲する。

- サブコマンドはargparse subparsers (`required=True`) で明示必須とする
- 実行アーカイブは既定有効、`--no-archive` または `archive = false` でオプトアウト可
- アーカイブ保存先はXDG準拠のユーザーキャッシュに集約（プロジェクトローカルには作らない）
- run_idはULID形式（26文字Crockford Base32）
- 自動クリーンアップは3軸（世代数 / サイズ / 期間）で古い順に削除
- Python系ツールは `pyfltr[python]` オプショナルグループに集約（細粒度extrasは不採用）
- MCPサーバーは本体同梱 (`pyfltr mcp` サブコマンド)、`mcp` / `python-ulid` / `platformdirs` は本体必須依存
- severity正規化は `error` / `warning` / `info` の3値、未対応ツールは `None` 省略
- rule_urlは既知ツール (ruff / pylint / pyright / mypy / shellcheck / eslint / markdownlint / textlint) のみ提供
- smart truncationはJSONL側でdiagnostic上限 + tool.message長を制限、truncate分はアーカイブへ全文保存
- ファイルhashキャッシュは既定有効 (`--no-cache` または `cache = false` でオプトアウト)、ファイル間依存を持たず設定もCWDのみで解決するlinter（textlint）のみ対象。既定保持時間12時間（`cache-max-age-hours`）、期間軸のみのシンプルなクリーンアップ
- `--fail-fast` 成立のため `execute_command` のsubprocess実行経路をPopenに一本化する
- MCPツール範囲は読み取り系（`list_runs` / `show_run` / `show_run_diagnostics` / `show_run_output`）と実行系（`run_for_agent`）の計5件
- MCPライブラリはFastMCP（`mcp.server.fastmcp.FastMCP`）を採用。型ヒントからinput/output schemaが自動生成される
- MCPツールの公開名はアンダースコア形式（`list_runs`等）。対応CLIのハイフン形式（`list-runs`等）とは表記が異なる
- `run_for_agent` MCPツールは実行アーカイブを強制有効化し、`run_id`を戻り値に含む契約を保証する
- `run_pipeline()`の戻り値を`tuple[int, str | None]`に拡張し、MCPツールから採番済み`run_id`を取り出す経路を確保する
- `retry_command`のターゲットは当該ツールで失敗したファイルのみに絞る（パートG A案）。絞り込み結果が空の場合はターゲット位置引数を空のまま出力する（pytestのような全体指定で意味を持つツールに対応するため）
- `--only-failed`の絞り込み結果は`dict[str, list[pathlib.Path] | None]`でツール別に保持する。`None`は診断なしフォールバック、空リストはtargets交差なしで除外を意味する
- `--only-failed`で直前runが無い・失敗ツールが無い・全ツールで交差なしの場合はメッセージを出して`rc=0`で成功終了する（反復CI運用を阻害しないため）
