# パート E: 詳細参照サブコマンド

v3.0.0で追加する `list-runs` / `show-run` サブコマンドの設計判断と仕様。
実行アーカイブ（パートB）に保存された情報をCLIから参照する手段を提供する。

## 目的

実行アーカイブ ([実行アーカイブ](実行アーカイブ.md)) に保存された過去runの参照手段をCLIで提供する。
JSONLのsmart truncationで削られた診断や、ツール生出力の全文を事後参照したいケースに対応する。
パートFのMCPサーバーで提供する詳細参照ツールも、本サブコマンドと同じ読み取り経路 (`archive.ArchiveStore`) を共有する。

## 提供サブコマンド

### list-runs

保存済みのrunを新しい順で一覧表示する。

主な要件:

- 既定で表示件数を制限する（例: 直近20件）。`--limit` で変更可能
- 1行 = 1 run。`run_id` ・ `started_at` ・ `exit_code` ・ `commands` ・ `files` を表示
- 出力フォーマットを `--output-format` で選択可能 (text / json)

### show-run

指定runの詳細を表示する。

主な要件:

- 必須引数: `<run_id>`（前方一致や `latest` などのエイリアスはDesignフェーズで検討）
- 既定ではmeta + ツール一覧 (各ツールの `status` ・ `has_error` ・ `diagnostics` 件数) を表示
- `--tool <name>` で特定ツールに絞り込み、tool.json + diagnostics.jsonlを全件表示
- `--output` で当該ツールのoutput.log全文を表示するモード
- 出力フォーマットを `--output-format` で選択可能 (text / json / jsonl)

## スコープ

| 項目 | 内容 |
| --- | --- |
| 対象モジュール | `pyfltr/main.py` (サブパーサー追加) ・ 新規 `pyfltr/show_run.py` 等 (実装配置は Design で確定) |
| 既存基盤 | `pyfltr/archive.py` の読み取り API (`list_runs` / `read_meta` / `read_tool_meta` / `read_tool_output` / `read_tool_diagnostics`) を流用 |
| CLI追加 | `pyfltr list-runs` ・ `pyfltr show-run` |
| 設定キー追加 | なし (アーカイブの保存先は既存設定を利用) |
| 出力形式 | text（人間向け）・ json（単発dict）・ jsonl（1件1行ストリーム） |
| MCP との関係 | パート F の詳細参照ツールが本サブコマンドの読み取り経路を共有する |

## 受け入れ基準

- `pyfltr list-runs` で過去run一覧が新しい順に表示される
- `pyfltr list-runs --limit N` で件数を制御できる
- `pyfltr show-run <run_id>` でmetaとツール別サマリが表示される
- `pyfltr show-run <run_id> --tool <name>` でツール別のdiagnostics全件が表示される
- `pyfltr show-run <run_id> --tool <name> --output` でツールの生出力（`output.log` 全文）が表示される
- `--output-format=json` で機械可読な単発出力、`--output-format=jsonl` で1件1行のストリーム出力が得られる
- 存在しないrun_id ・ tool名指定時は分かりやすいエラーメッセージで終了コード非ゼロを返す
- アーカイブ無効時（`--no-archive` で過去に書き込まれていない場合）でも `list-runs` は空リスト返却で正常終了する

## 主要設計判断

- 実装配置は `pyfltr/runs.py` に集約する。`main.py` は既存の `generate-config` ・ `generate-shell-completion` と同じ「非実行系サブパーサー」としてサブパーサー登録とディスパッチのみを行い、出力ロジックは持たない。
- 読み取り経路は `archive.ArchiveStore` の既存APIを直接利用する。対象は `list_runs` / `read_meta` / `read_tool_meta` / `read_tool_output` / `read_tool_diagnostics`。
- `load_config()` は呼ばない。キャッシュルートの上書きは環境変数 `PYFLTR_CACHE_DIR` のみで、サブコマンドの依存を最小化できる。
- 「指定runの実保存ツール一覧」の取得用に `ArchiveStore.list_tools(run_id)` を新規追加する。`meta["commands"]` は実行予定のリストで、`--fail-fast` 中断や `skipped` で実際には保存されなかったツールを含みうる。`tools/` ディレクトリ走査が実保存ツールのSSOTとなる。
- アーカイブの保存キーは `tools/<sanitize(tool)>/` 固定のため、同一ツール名のfixステージと通常ステージは通常ステージで上書きされる。`show-run` は各ツールの最終保存結果のみを参照可能で、ステージ別保存への拡張は本パートのスコープ外とする。
- `run_id` 指定は完全一致に加えて前方一致と `latest` エイリアスを許容する。前方一致で複数該当した場合は曖昧と判定し終了コード1を返す。
- `--output-format` は `text` / `json` / `jsonl` の3種とする。`sarif` / `github-annotations` は実行結果向け形式で詳細参照とは目的が異なるため対象外。
- `text` 出力は `list-runs` が固定幅テーブル、`show-run` が行形式 `key: value`。`json` は単発dict、`jsonl` は1件1行ストリーム。
- `json` / `jsonl` モードのみroot loggerを抑止してstdoutを構造化出力で専有する。`text` モードではloggerへの影響は無く、`print` が直接stdoutへ出る。
- 既定表示はmeta + ツール別サマリ (`status` / `has_error` / `diagnostics` 件数) のみ。diagnostics全件表示は `--tool <name>`、生出力全文は `--tool <name> --output` で明示する。
- 終了コードは「アーカイブ未存在の `list-runs`」のみ0（空リスト扱い）。存在しない `run_id` ・ 前方一致での複数該当 ・ 存在しない `--tool` 指定はいずれも1を返す。

## 却下した代替案

- `main.py` 内に `list-runs` / `show-run` を直書きする案 — 出力フォーマット分岐を含み肥大化する。既存の `sarif_output.py` ・ `github_annotations.py` 等と粒度を揃えて分離する方が保守的。
- ツール一覧の取得を `meta["commands"]` から行う案 — `--fail-fast` 中断や `skipped` で実際にはアーカイブされなかったツールを含みうるため、実保存ツールのSSOTとしては使えない。
- アーカイブ保存キーを `tools/<tool>__<stage>/` 形式に変えてステージ別保存にする案 — パートB成果物への破壊的変更となる。
    `write_tool_result` ・ `tool.json` ・ `archive_test.py` 等への影響が大きく、本パートの目的からも外れる。必要になった時点で別パートで検討する。
- `run_id` を完全一致のみとする案 — ULID 26文字を毎回手入力させるUXは現実的でない。
- `show-run` の既定表示にdiagnostics冒頭数件を含める案 — 情報量制御が困難で、`--tool` 指定時の挙動と重複する。
- `text` 出力も `logger.info` 経由にする案 — `--verbose` の影響を受けてサブコマンド出力としての一貫性が崩れる。`print` 直接が素直。
- `--output-format` に `sarif` / `github-annotations` を含める案 — 実行結果向け形式で詳細参照とは目的が異なる。
- `load_config()` を呼んで設定経由でキャッシュルートを上書きする案 — `pyproject.toml` に対応キーが存在せず、現状の環境変数経路で完結する。将来必要になった時点で追加すればよい。
