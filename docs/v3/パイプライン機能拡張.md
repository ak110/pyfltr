# パート D: パイプライン機能拡張

v3.0.0で追加したパイプライン側の機能拡張の設計判断と実装方針。

## 目的

エージェント連携時の待ち時間と無駄な再計算を削減するため、パイプライン本体に次の3点を導入する。

- `--fail-fast`: 1ツールでもエラーが発生した時点で残りの実行を中断し、結果を返す
- ファイルhashベースのスキップキャッシュ: 同じ入力に対する再実行コストを削減する
- `execute_command` のPopen一本化: 早期中断・将来のリアルタイム出力の土台となる実行基盤の統一

## スコープ

| 項目 | 内容 |
| --- | --- |
| 対象モジュール | `pyfltr/command.py` / `pyfltr/main.py` / `pyfltr/cli.py` / `pyfltr/ui.py` / `pyfltr/config.py` / `pyfltr/llm_output.py` / 新規 `pyfltr/cache.py` |
| CLI追加 | `--fail-fast` / `--no-cache` |
| 設定キー追加 | `cache` / `cache-max-age-hours` |
| 出力形式への影響 | JSONL `tool` レコードに `cached` / `cached_from` を条件付き付与、`summary.skipped` にfail-fast中断対象を計上 |
| アーカイブとの関係 | キャッシュヒット時はアーカイブ書き込みをスキップし、ソース `run_id` を `cached_from` で参照させる |

## --fail-fast

いずれかのツールが `has_error=True` で完了した時点で、残りのジョブを中断する。

- 早期中断の判定単位: ツール完了時の `CommandResult.has_error`
- formatterの `formatted` ステータスはfailureに含めない（中断契機にしない）
- 未開始ジョブ: `concurrent.futures.Future.cancel()` でsubmit取消
- 起動済みサブプロセス: `terminate()` → 最大5秒待機 → 生存していれば `kill()` でフォールバック
- JSONL出力: 中断されたツールは `status="skipped"` のtoolレコードを発行し、`summary.skipped` に計上する
- アーカイブ: 中断ツールのskipped結果も `archive_hook` へ渡すため、`finalize_run` が必ず呼ばれる経路になっている

## ファイルhashキャッシュ

同じ入力に対するツール再実行をスキップする仕組み。

### 対象ツール

ファイル間依存を持たず、設定ファイルもCWDでのみ解決するlinterが対象。具体的には `textlint` のみ。

対象判定は `CommandInfo.cacheable: bool` フィールドで明示する。新ツール追加時の判断ミスを防ぐため既定値は `False` とし、対象ツールのみ `True` を指定する。

以下はいずれも対象外。

- 書き込み型formatter: `ruff-format` / `prettier` / `shfmt` / `uv-sort`
- tester: `pytest` / `vitest`
- 依存型linter: `mypy` / `ruff-check` / `pylint` 等
- 外部参照linter: `shellcheck` / `actionlint`
- 階層構造に依存する設定ファイルを参照するlinter: `ec` / `markdownlint` / `typos`

### キャッシュキー

誤ヒットを防ぐため、次の情報すべてをsha256で連結する。

- ツール名
- 実効コマンドライン全体（`--{command}-args` 反映後のリスト）
- ステージ区別（`fix_stage` フラグの真偽）
- 構造化出力の設定値（`--human-readable` 反映後、ツールが構造化出力を持つ場合）
- 対象ファイル群のsha256（ファイル内容＋相対パス）
- ツール固有設定ファイル群のsha256（存在しない場合は空文字扱い）
- pyfltrのMAJORバージョン（メジャー更新で出力フォーマット変更時に旧キャッシュを無効化）

ツール本体のバージョンはキャッシュキーに含めない。短期破棄前提（既定12時間）で許容する方針。

### textlint の config_files

公式に自動読み込みされる設定ファイル候補を完全列挙する。

- `.textlintrc` / `.textlintrc.json` / `.textlintrc.yml` / `.textlintrc.yaml`
- `.textlintrc.js` / `.textlintrc.cjs`
- `package.json`（`textlint` フィールド）
- `.textlintignore`

### 外部ファイル参照引数の安全策

`--{command}-args` に `--config` / `--ignore-path` を含む場合、当該実行ではキャッシュを無効化する（書き込みも読み出しもスキップ）。指定されたパスを動的に解釈する複雑さを避けるための安全側の実装。

### 保存先とクリーンアップ

実行アーカイブと同じユーザーキャッシュ配下に `<cache_root>/cache/<tool>/<hash>.json` として保存する。

クリーンアップは期間軸のみ（既定 `cache-max-age-hours=12`）。サイズ・世代数の軸は採用しない（短期破棄前提でストレージ暴発リスクが小さいため）。

### ヒット時の挙動

- ツール実行をスキップして `CommandResult` を完全復元し、`cached=True` / `cached_from=<ソースrun_id>` を設定する
- アーカイブ書き込みは行わない（同じ結果を重複記録しない方針。ソースrunを `cached_from` で参照誘導する）
- JSONLのtoolレコードに `cached` / `cached_from` を出力する

## execute_command のPopen一本化

### 対象範囲

`_run_subprocess()` の `subprocess.run` 経路と `subprocess.Popen` 経路を統一する。

対象外:

- `_resolve_bin_commandline()` の `mise --version`
- `_filter_by_gitignore()` の `git check-ignore`
- `main.py` の `cls`/`clear`（パイプライン外）

### スレッドセーフ性

`_active_processes` への並列アクセスを `threading.Lock` で明示的に排他する。`--fail-fast` が別スレッドから `terminate_active_processes()` を呼ぶ際、プロセスリストの追加・削除と衝突しないための安全策。

### 早期中断API

`terminate_active_processes()` は、登録中のすべてのPopenに対して `terminate()` を送り、指定タイムアウト（既定5秒）待ってもなお生存しているプロセスには `kill()` をフォールバックで送る。Windowsでも `Popen.terminate()` は `TerminateProcess` 経由で動作する。

## 却下した代替案

- ファイルhashキャッシュを既定無効（オプトイン）にする → エージェント連携時のUXを毀損するため。実行アーカイブ（既定有効＋オプトアウト）と方針を揃える
- 書き込み型formatterをキャッシュ対象に含める → ヒット時にファイル書き換えがスキップされ、ファイル状態と結果が不整合になる
- testerをキャッシュ対象に含める → 対象ファイル以外のソース・設定・環境への依存があり、依存解析またはプロジェクト全体hashが必要で実装コストに見合わない
- 依存型linterをキャッシュ対象に含める → import先や型情報のキャッシュが必要で、対象ファイル単独のhashでは整合性を保てない
- `shellcheck` / `actionlint` をキャッシュ対象に含める → `source` 文やreusable workflowなど外部参照を含みうるため安全側で除外
- `ec` / `markdownlint` / `typos` をキャッシュ対象に含める → いずれも階層型の設定解決を行うため静的列挙では網羅できない
- ツール本体のバージョンをキャッシュキーに含める → `pnpx` / `mise @latest` 経由で実体が変わりうるが、キャッシュ保持期間を短く（既定12時間）することで実害を許容する
- キャッシュヒット時もアーカイブを書き込む → `show-run` / `list-runs` での閲覧性向上と引き換えにキャッシュ機構の軽量化が損なわれる。`cached_from` 参照誘導で代替
- `--fail-fast` の中断単位をファイル単位にする → ツール並列実行モデルと相性が悪く、途中強制終了は誤検出を生みやすい
- cli.py / ui.pyの並列実行ロジックを本パートで `executor.py` に統合する → パートDのスコープが膨張し、回帰リスクが上がるため別作業に分離
