# パート A: 依存整理・破壊的変更

v3.0.0で実施する破壊的変更群の設計判断と実装結果。

## 削除したツール

以下5ツールおよびその派生設定キーをすべて削除した。

- `pyupgrade`
- `autoflake`
- `isort`
- `black`
- `pflake8`

削除理由はruffへの統合で代替可能となり、プリセット `20250710` 以降は既定で無効化されていたため。
保守対象として長期間コードベースに残り続ける負債を断ち切る意図で、v3.0.0の破壊的変更に合わせて一括削除とした。

影響範囲:

- `BUILTIN_COMMANDS`・`DEFAULT_CONFIG`・`PYTHON_COMMANDS`・`aliases` から該当エントリを除去
- `command.py` の2段階実行分岐 (`--check` 付与の往復) を撤去
- `pyproject.toml`の本体必須依存から`autoflake` / `black` / `isort` / `pyupgrade`を除去。
  あわせて`pyproject-flake8` / `flake8-bugbear` / `flake8-tidy-imports`も除去

## 削除したプリセット

プリセット `"20250710"` を削除した。
削除5ツールを `False` にする役割しか持たなかったため、v3で該当ツール自体を削除した時点で内容を失う。

旧ユーザーが `preset = "20250710"` を設定していた場合、`load_config` が案内付きの `ValueError` を送出する。メッセージに `"削除"` と移行先 (`"latest"` 等) を含める。

## Python 系ツールの opt-in 化

v2系までは`tool.pyfltr`セクション未定義時にPython系ツールが既定で有効化される設計だった。
対象ツールは`mypy` / `pylint` / `pyright` / `ty` / `pytest` / `ruff-format` / `ruff-check` / `uv-sort`。
非Pythonプロジェクトで意図しないツール実行を招くため、v3で既定値をすべて `False` に変更した。

有効化手段:

- `python = true`を指定すると`PYTHON_COMMANDS`をまとめて有効化する
- `mypy = true`のように個別に有効化する方法もある
- 両方を指定した場合、後者が前者に優先する

`PYTHON_COMMANDS` の構成要素:

- `ruff-format`
- `ruff-check`
- `mypy`
- `pylint`
- `pyright`
- `ty`
- `pytest`
- `uv-sort`

## 依存パッケージの整理

本体必須依存を骨組み・共通ツール連携・run_id生成・MCPサーバー同梱に限定した。

v3以降の本体必須依存 (`[project.dependencies]`):

- `mcp>=1.0` — MCPサーバー実装
- `natsort>=8.4.0` — ファイルパス自然順ソート
- `platformdirs>=4.0` — OS横断キャッシュディレクトリ解決
- `python-ulid>=3.0` — run_id生成 (ULID)
- `pyyaml>=6.0.3` — pre-commit設定読み取り
- `textual>=8.0.0` — TUI

Python系ツールは `[project.optional-dependencies].python` へ移動した。

```toml
[project.optional-dependencies]
python = [
    "dill>=0.3",
    "mypy>=1.0",
    "pre-commit>=4.5.1",
    "pylint-pydantic>=0.4.1",
    "pylint>=3.0",
    "pyright[nodejs]>=1.1.403",
    "pytest-asyncio>=1.0.0",
    "pytest>=7.0",
    "ruff>=0.12.1",
    "ty>=0.0.26",
    "uv-sort>=0.5",
]
```

ユーザーはPythonプロジェクトで従来通りの利用感を再現するとき、`pip install pyfltr[python]` または `uv add 'pyfltr[python]'` で依存を追加する。

pyfltr自身の開発 (`[dependency-groups].dev`) では `"pyfltr[python]"` を依存に含め、自身のextrasを通じてPython系ツールを導入する。

## サブコマンド必須化

v2系までは引数なし実行時に `ci` サブコマンドとして扱うフォールバックがあった。
argparse subparsers機構に刷新したv3では、サブコマンドを明示的に指定しないとエラーになる。

刷新後の構造:

- `argparse.ArgumentParser` トップレベル + `add_subparsers(required=True)`
- 実行系サブコマンド (`ci` / `run` / `fast` / `run-for-agent`) は共通オプションを `parents=[common]` で継承する
- サブコマンド別の既定値（`exit_zero_even_if_formatted` / `commands` / `output_format` / `include_fix_stage`）は
  `_apply_subcommand_defaults`で手動注入する

`set_defaults`経由での注入を避けたのは、argparseの既知挙動を回避するためである。
`parents=[common]`で共通化されたnamespaceを通じて他サブパーサーのdefaultも書き換えてしまう問題が、共通親パーサーを継承している際のみ発生する。

### サブコマンド一覧

- `ci`: CIモード。formatterの書き換えも失敗扱い。fixステージ無効
- `run`: 通常実行。formatterの書き換えは成功扱い。fixステージ有効
- `fast`: 高速ツールのみ (`--commands=fast`)。fixステージ有効
- `run-for-agent`: LLMエージェント向け。`--output-format=jsonl` 既定。fixステージ有効
- `generate-config`: 設定雛形を標準出力へ書き出す
- `generate-shell-completion <shell>`: 補完スクリプトを標準出力へ書き出す

## 関連テスト

破壊的変更に合わせて以下のテストを更新した。

- `tests/config_test.py`: プリセット検証から5ツール項目を除去、`python` 既定値変更の反映、opt-in挙動の新規テスト追加、削除プリセット/削除ツール設定キーのValueErrorテスト
- `tests/main_test.py`: `_parse_subcommand` / `_build_effective_args` のテスト除去、サブコマンド明示必須化に伴う書き換え、アーカイブ連携テスト
- `tests/executor_test.py` / `tests/cli_test.py` / `tests/command_test.py` / `tests/output_format_test.py`: 削除ツール参照をruff系へ置換
- `tests/shell_completion_test.py`: `_parse_subcommand` の除去反映、サブパーサー対応
- `tests/archive_test.py`: 新規作成（パートB）

## 却下した代替案

- Python系ツールを完全別パッケージ (`pyfltr-python`) として分離: リポジトリ・リリース・バージョン整合の複雑度が増し、ユーザー体験もextras方式より劣るため却下
- 5ツール削除を段階的（v3でdeprecation警告のみ、v4で削除）にする: コード負債を延命する。破壊的変更の機会にまとめて整理するのが合理的
- サブコマンド必須化をしない互換モード保持: 設定分岐で保守コストが増える。破壊的変更の趣旨に反する
