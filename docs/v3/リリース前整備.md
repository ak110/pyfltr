# リリース前整備

v3.0.0のパートA〜G実装完了後に取り込む統合整備。
コード構造リファクタ・API設計改善・運用改善の三系統をまとめて実施し、完全な状態でv3.0.0のタグ付けを行う。

## 恒常配置の対応先

v3.0.0全体で恒常配置（`docs/features/`・`docs/topics/`）は未整備のため、本整備も開発中配置のみで完結する。
v3リリース完了後に`spec-driven-promote`スキルで恒常配置を整備する際、本ドキュメントの各系統は次の候補ドキュメントへ分散昇格する。

- コード構造リファクタ（`pyfltr/paths.py` / `pyfltr/retry.py` / `pyfltr/only_failed.py`）→ 内部設計ドキュメントとして恒常配置（未確定。v3リリース時点で判断）
- API設計改善（`--from-run` / `ToolTargets`）→ `docs/features/再実行支援.md` へ昇格（パートGと統合）
- 運用改善（run_idテキスト表示・mkdocs strict対応）→ `docs/guide/usage.md` 等のユーザー向けガイドへ反映

## 目的と対応方針

本整備は三系統の課題を解消し、v3.0.0を完成形で公開することを目的とする。
各系統の目的と対応方針を次に示す。

### コード構造リファクタ

パートA〜Gの実装積み重ねにより`pyfltr/main.py`が1100行超に肥大化し、`# pylint: disable=too-many-lines`を付与したまま放置している状態を解消する。
retry系ヘルパー・`_apply_only_failed_filter`・パス正規化の重複箇所をそれぞれ専用モジュールへ分離する。

`pyfltr/cli.py`と`pyfltr/ui.py`はfix→formatter→linter-testerの3ステージ分岐がほぼ同一の構造を持ち、`_make_skipped_result`・futureキャンセルループが二重実装されている。
`_make_skipped_result`と`cancel_pending_futures`の重複は`pyfltr/stage_runner.py`へ集約する。
ただし`call_from_thread`が絡む残余重複は構造が異なるため完全共通化せず、`# pylint: disable=duplicate-code`は理由コメント付きで維持する。

新規モジュールとして以下を作成する。

- `pyfltr/paths.py`: `normalize_separators(path)`ヘルパー（`str(path).replace("\\", "/")`の共通化）
- `pyfltr/retry.py`: retry系ヘルパー5関数（`detect_launcher_prefix`等）と`_VALUE_OPTIONS`定数を`main.py`から移動する
- `pyfltr/only_failed.py`: `apply_filter()`（旧`_apply_only_failed_filter`を5サブ関数に分割）と`ToolTargets` dataclassを定義する
- `pyfltr/stage_runner.py`: `make_skipped_result()` / `cancel_pending_futures()`を`cli.py`・`ui.py`から抽出して共通化する

`pyfltr/main.py`の`# pylint: disable=too-many-lines`撤去が完了の指標となる。

### API設計改善

`--only-failed`の内部値型（`dict[str, list[pathlib.Path] | None]`）を専用dataclass`ToolTargets`で置き換え、`None`/空リストの二状態の意味を型で明示する。
`--from-run`フラグを新設し、`--only-failed`の参照対象runをアーカイブの前方一致・`latest`エイリアスで明示指定できるようにする。

`ToolTargets` dataclassは`mode: Literal["fallback", "files"]`と`files: tuple[pathlib.Path, ...]`を持つfrozen dataclassとする。
クラスメソッド`fallback_default()` / `with_files(files)`と、インスタンスメソッド`resolve_files(all_files)`を提供する。

`--from-run`は`--only-failed`との併用のみを受け付け、単独指定はargparse側でエラーとする。
`--from-run`の値および`--only-failed`フラグは`retry_command`へ伝播させない。
生成する`retry_command`は「当該ツール＋失敗ファイル」に固定されているため、アーカイブ参照フラグを引き継ぐと再実行時に古いrunを暗黙参照し続けるリスクがあるため。

### 運用改善

テキスト出力経路でrun_idが表示されない問題を解消する。
実行開始時とsummary末尾の2箇所に追加し、`pyfltr show-run latest`への誘導文言を添える。
`mkdocs build --strict`実行時に日本語アンカーリンクの不一致によるINFO通知が3件発生する問題を、リンク参照側を実際の生成アンカーIDへ修正することで解消する。
spec-drivenで生成する`*.research-*.md`が`.gitignore`に登録されていない漏れも解消する。

mkdocsのアンカー不一致は、リンク参照側を実際のHTMLアンカーIDへ書き換える最小対応で済ませる（影響範囲が小さいため）。
`*.research-*.md`は`.gitignore`の`*.review-*.md`の直後に追加する。

## 主要設計判断

### `ToolTargets` dataclassで値型の意図を明示する

旧`dict[str, list[pathlib.Path] | None]`では`None`と空リストの違いがコードを読むだけでは不明瞭だった。
`ToolTargets`の`mode`属性（`"fallback"` / `"files"`）で二状態を型として区別し、`resolve_files(all_files)`で実際の対象ファイルリストを取得する統一経路を持たせる。

skip状態（ツールを再実行対象から除外）は`commands`絞り込みの段階で除外されるため、`ToolTargets`の値域に含めない。これにより`dict`の値型がシンプルになる。

### `main.py`の分割方針

関数・定数の移動先を3モジュール＋1共通ヘルパーに整理する。

- `pyfltr/retry.py`: retry系ヘルパー（`build_retry_args_template`等）と`_VALUE_OPTIONS`
- `pyfltr/only_failed.py`: `apply_filter()`と`ToolTargets`
- `pyfltr/paths.py`: `normalize_separators()`
- `pyfltr/stage_runner.py`: cli/ui共通ヘルパー（`make_skipped_result()` / `cancel_pending_futures()`）

`run_pipeline`（213行）はパイプライン中核として許容範囲のため、分割対象から外す。

### cli/uiは完全共通化ではなくヘルパー化に絞る

`cli.py`の`call_from_thread`なし・`ui.py`のRich UIへの`call_from_thread`埋め込みという構造差があるため、ステージランナー全体の完全共通化は実装が複雑になる。
`make_skipped_result()`・`cancel_pending_futures()`の2関数をSSOT化するにとどめ、ステージループ本体は各ファイルに残す。

### `--from-run`のrun_id解決に既存`resolve_run_id()`を再利用する

`pyfltr/runs.py`の`_resolve_run_id()`を`resolve_run_id()`としてpublic化し、`only_failed.apply_filter()`から呼び出す。
同関数は前方一致・`latest`エイリアス・曖昧prefix判定を既に提供している。
`pyfltr/mcp_.py`がprivateメソッドを`# noqa: SLF001`で参照している箇所も合わせて追従する。

### mkdocsはリンク参照側を修正する最小対応にとどめる

`mkdocs-autorefs`プラグインはAPI DOCユース向けで使用ガイド型ドキュメントへの効果が限定的であり、導入コストに見合わない。
全見出しへの`{#anchor-id}`記法の機械的な追加は全docs横断の修正になりコストが過大である。
リンク参照側（`docs/guide/usage.md`内の3箇所）を実際の生成アンカーIDへ書き換えることで影響範囲を最小化する。

## 却下した代替案

- **`--from-run`を`--only-failed`なしで単独利用可能にする案**: `--from-run`単独では`diagnostic`参照は行われず意味を持たないため却下
- **`--from-run`値をretry_commandへ伝播させる案**: 再実行時に意図せず古いrunを参照し続けるリスクがあるため却下
- **`ToolTargets`にskip状態（3状態）を含める案**: `commands`絞り込みで除外するためdictには含めず、値型をシンプルに保つ
- **cli/uiの完全な共通ステージランナー導入**: `call_from_thread`/lock取得タイミング差で実装が複雑化するため、共通ヘルパー化に絞る
- **mkdocs-autorefsプラグイン導入**: 使用ガイド型ドキュメントへの効果が限定的で導入コストに見合わない
- **全見出しへの`{#anchor-id}`記法の機械的な追加**: 全docs横断の修正でコスト過大
- **本整備を5テーマ別の計画ファイルに分割する案**: ユーザーが一括対応を明示したため不採用
- **v3.1.0以降への段階対応**: ユーザー指示で前倒しのため不採用
