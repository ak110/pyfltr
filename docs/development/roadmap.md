# ロードマップ

本ページはpyfltr本体の保守・貢献に関わる開発者を対象に、エージェント連携の拡張候補を整理したもの。
いずれの項目も現時点で採否未決であり、本ページは後日の議論・判断のための土台として位置づける。

各候補は次の4観点で整理する。

- 目的: 何を解決したいか
- 実装規模感: 新規モジュール追加や既存コードへの影響の大小
- 得られる価値: エージェント連携・利用体験の改善点
- 懸念: 実装/運用上のリスクや設計上の論点

## 候補一覧

### 診断のseverity優先度付け・集約

- 目的: `diagnostic`レコードに`severity`相当の情報を加え、エージェントがエラー / 警告 / 情報を区別できるようにする
- 実装規模感: 中。各ツールの`error_parser`へseverity抽出を追加し、JSONLスキーマへフィールドを追加する
- 得られる価値: エージェントが「まずerror」「次にwarning」と段階的に修正対象を選べ、トークン消費の最適化に寄与する
- 懸念: ツールごとにseverity表現が異なり、正規化マッピングの設計で合意形成が必要。既存JSONLスキーマの後方互換方針も論点

### ツール失敗時のリトライ指示出力埋め込み

- 目的: エージェント向けに「次にどのコマンドを再実行すべきか」を`tool`/`summary`レコードへ提案として埋め込む
- 実装規模感: 小〜中。既存`tool`レコードの追加フィールドとして提案コマンド文字列を書き込む
- 得られる価値: エージェントがワンショットで全体実行した後の絞り込みに迷わず進める
- 懸念: 提案内容がpyfltr側の暗黙の方針になるため、複数エージェント間での解釈ぶれを避ける定義が必要

### diagnosticへのrule doc URL付与

- 目的: `diagnostic`レコードに`rule_url`等を加え、ルールIDからドキュメントへのリンクを提供する
- 実装規模感: 中。ツールごとにdoc URLのテンプレートを持つマッピングが必要
- 得られる価値: エージェントがWeb検索なしにルールの意図を確認でき、修正品質が上がる
- 懸念: ツール側のdocサイト構成変更に追従するメンテコスト。全ツール網羅は現実的でないため対象ツールの選定が論点

### 出力のsmart truncation / 要約

- 目的: 長大な出力（数千行の診断など）をエージェントのコンテキスト節約のため安全に切り詰める・要約する
- 実装規模感: 中〜大。truncationポリシーの設計、summaryとの整合性確保、元データへのアクセス手段が必要
- 得られる価値: トークン消費の大幅削減と、大規模リポジトリでの実用性向上
- 懸念: 要約により診断の取りこぼしが起きるリスク。切り詰め基準の自動判定は難しく、閾値設定のUXが論点

### SARIF / GitHub Annotation出力対応

- 目的: JSONLに加えてSARIFまたはGitHub Actions向けannotation形式を出力可能にし、CI連携を容易にする
- 実装規模感: 中。新しい`--output-format`値の追加と、各ツールの診断からのマッピング
- 得られる価値: GitHubのPRコメント欄に自動でannotationを出せる、セキュリティ系ツールとの連携が進む
- 懸念: SARIFは仕様が大きく、全ツール対応のコストが高い。最小限のサブセットで妥協する設計が論点

### `--fail-fast`

- 目的: 最初の失敗で以降のツール実行を打ち切り、エージェントの試行サイクル時間を短縮する
- 実装規模感: 小。既存パイプラインに早期打ち切りフラグを追加する
- 得られる価値: 時間のかかる複数ツール実行での初回フィードバックが速くなる
- 懸念: 並列実行中の打ち切りタイミング・未完了ツールの結果記録・JSONLの整合性確保が論点

### ファイルhashベースのスキップキャッシュ

- 目的: 前回実行時から変更のないファイルに対するチェックをスキップし、大規模リポジトリでの実行時間を削減する
- 実装規模感: 大。キャッシュストア、無効化条件、ツールごとの副作用分類が必要
- 得られる価値: CI/ローカル双方で繰り返し実行の時間を大幅短縮できる可能性がある
- 懸念: ツール設定変更やツール本体バージョン変更の検知漏れでfalse positiveが出る。キャッシュ不整合時の復旧UXが論点

### MCPサーバー化

- 目的: Model Context Protocol（MCP）サーバーとしてpyfltrを公開し、Claude Codeなどから構造化ツールコールで呼び出せるようにする
- 実装規模感: 大。新規サーバー実装、既存CLIとの責務分担、バージョン管理方針が必要
- 得られる価値: エージェントが`Bash`経由ではなく専用ツールコールでpyfltrを操作でき、出力解釈のロバスト性が向上する
- 懸念: MCP仕様の進化に追随するコスト。既存CLIとの機能重複・互換性をどう保つかが論点
